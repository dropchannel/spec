# ADR-XXXX: Extract Ripple as a First-Class Protocol Spec

## Context

DropChannel defines protocol version parity as the compatibility contract between nodes and clients. Nodes must agree on which versions of each protocol they implement; the version of the language-specific implementation (e.g., `dropchannel-py` vs `dropchannel-ru`) is not itself the compatibility boundary.

A system-level protocol currently exists in `dropchannel-py` under the name **heartbeat**. It is the protocol by which rafts emit system-generated payloads that propagate outward through a waterway's topology in concentric, successively stacked hops. These payloads create visibility for end clients into the waterway they are interacting with — conveying what rafts and clients are active, what state each participant is in, and the operational condition of the far end of the waterway at any given moment. It functions as a combination of health signal, presence indication, and ambient status propagation. It is the waterway making itself legible to its endpoints.

Because protocol parity governs compatibility, this protocol carries the same versioning obligations as transport protocols (Winch, Conveyer, Ring, Piston). Leaving its spec embedded within a single implementation repo (`dropchannel-py`) is inconsistent with that governance model: it makes the Python implementation the implicit source of truth for behavior that all future implementations must match.

This protocol has a distinctive payload topology that merits a first-class name: payloads originate within the waterway (system-generated by rafts) and propagate outward in a concentric ripple pattern. The topology is inward-to-outward, bounded by the waterway. The protocol is accordingly named **ripple**, and its repo **ripple-protocol**.

Ripple is explicitly scoped to system-generated payloads. Use for client payloads is out of scope and unsupported. This is an intentional design boundary, not an oversight; lifting it in future would require a versioned, ADR-backed decision.

## Decision

### 1. New protocol spec repo

A new protocol spec repo, `ripple-protocol`, will be created in the `dropchannel` org. It follows the same versioning and repo conventions as existing protocol spec repos (`winch-protocol`, `ring-protocol`, etc.), including a `history/` directory for versioned spec snapshots and the draft/stable tagging model defined in ADR-XXXX.

### 2. Spec extraction

The existing heartbeat spec will be extracted from `dropchannel-py` and migrated into `ripple-protocol` as the authoritative spec. The heartbeat spec content in `dropchannel-py` should be updated to reference the canonical spec repo rather than duplicate it.

### 3. Implementation remains in language repos

Protocol implementation remains in language-specific repos (`dropchannel-py`, and future `-ru` etc.). The spec repo governs behavior; the implementation repos govern code.

### 4. Protocol versioning

Ripple protocol versions cover the full stack: propagation mechanics, payload format, and encryption envelope version together as a unit. A change to any layer yields a version bump. Changelog entries should note which layer was affected. This is consistent with the versioning model established for transport protocols in ADR-XXXX.

Propagation mechanics are an explicit part of the versioned wire surface. This includes hop count and termination rules, per-hop re-emission behavior, deduplication, and trigger conditions. This is less obvious than format or encryption versioning and is called out explicitly here to avoid ambiguity.

### 5. Channel naming convention

Ripple is specific to a waterway and has no meaningful use outside of one. Accordingly, ripple channels do not use the generic `{protocol}--{name}` convention defined in ADR-XXXX. Instead, ripple channel names are derived directly from their parent waterway:

```
{waterway-protocol}-ripple--{waterway-name}
```

Where `{waterway-protocol}-ripple` is the protocol identifier and `{waterway-name}` is the user-selected name of the parent waterway.

#### Examples

| Parent waterway | Ripple channel |
|---|---|
| `riverway--chat` | `riverway-ripple--chat` |
| `tideway--cpu-state` | `tideway-ripple--cpu-state` |
| `riverway--orders` | `riverway-ripple--orders` |

This convention makes association with the parent waterway unambiguous, eliminates any possibility of name collision between ripple channels, and is consistent with the physical metaphor (a riverway has ripples; a tideway has ripples).

The waterway name component of the ripple channel name must exactly match the user-selected name of the parent waterway. A ripple channel with no corresponding waterway is invalid.

## Consequences

- `ripple-protocol` becomes the source of truth for ripple behavior. `dropchannel-py` (and future implementation repos) are consumers of that spec, not owners of it.
- Future implementations (e.g., `dropchannel-ru`) must implement ripple to the same version parity standard as transport protocols.
- The `{waterway-protocol}-ripple--{waterway-name}` naming convention is binding for all ripple channel names org-wide. The `PROTOCOL_REGISTRY` dispatch logic must recognise this pattern.
- Restricting ripple to system-generated payloads is an explicit design boundary. Lifting this restriction in future would be a versioned, ADR-backed decision.
- This extraction has no effect on client payload security. Client payloads transmitted via transport protocols (Winch, Conveyer, Ring, Piston) remain encrypted under AES-256-GCM; that encryption model is unaffected by the introduction or governance of ripple. A non-conforming raft could theoretically use ripple to write data to channel storage, but this does not represent a meaningful change to the client payload threat model: a non-conforming raft already has the ability to write to arbitrary storage locations. Ripple names an existing structural capability; it does not introduce a new exfiltration vector.
- Ripple payloads are currently unencrypted. This is a pre-existing condition and a known open concern; it is not introduced by this ADR. Encryption of ripple payloads is a separate future consideration.
