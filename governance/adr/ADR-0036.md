# ADR-0036: Agent merged into dropchannel-node rather than separate package

**Status:** Accepted
**Date:** 2026-03-17
**Deciders:** Paul Czajka

---

## Context

v0.9 introduces the Agent — a host-level supervisor process that spawns and manages
Worker subprocesses, each of which runs the existing `dropchannel-node` protocol
forwarding logic. The Agent requires a new entry point (`dc-agent`) and several new
modules (config loading, log aggregation, worker supervision, telemetry writing).

The question was where this code should live: as a new sixth package
(`dropchannel-agent`), or as a submodule of the existing `dropchannel-node` package.

The existing package split has four packages with clear, principled boundaries:

| Package | Boundary rationale |
|---------|-------------------|
| `dropchannel-channel` | Provider interface and implementations; no protocol logic |
| `dropchannel-endpoint` | Exclusively holds crypto; the only package that may import `cryptography` |
| `dropchannel-node` | Protocol dispatch and forwarding; crypto-blind by enforced package boundary |
| `dropchannel-server` | Network-accessible relay; depends on channel only |

---

## Decision

Merge the Agent into `dropchannel-node` as an `agent/` submodule. Do not create a
separate `dropchannel-agent` package.

---

## Rationale

**The deployment unit is the same.** The Agent's sole purpose is to manage
`dropchannel-node` Workers. You would never install one without the other. A
separate package would introduce an artificial split between code that is
operationally and conceptually unified.

**The crypto isolation argument does not require separation.** The existing package
boundaries exist to enforce crypto isolation — specifically, to prevent `dropchannel-node`
from importing `cryptography`. The Agent is also crypto-blind. It never touches keys
or plaintext. Merging it into `dropchannel-node` does not weaken the isolation property;
the constraint applies equally to the new `agent/` submodule under the existing
`check_isolation.py` verification without any additional configuration.

**Subprocess invocation is simpler within the same package.** The Agent spawns Workers
as `python -m dropchannel.node`. If Agent and node were in separate packages, this
would be a cross-package subprocess invocation with an implicit runtime dependency
that is not expressed in `pyproject.toml`. Within the same package, the entry point
relationship is direct and unambiguous.

**The existing split would not have been made this way.** The four-package structure
reflects a deliberate split along crypto and functional boundaries. There is no
analogous boundary between Agent and node — the distinction is supervisor vs. worker,
not a security or capability boundary.

---

## Alternatives considered

### Separate `dropchannel-agent` package

A fifth package with its own `pyproject.toml`, `dc-agent` entry point, and
dependency on `dropchannel-node` and `dropchannel-channel`.

**Rejected because:** The deployment unit argument is strong. Operators install and
run `dc-agent` on the same machine as `dropchannel-node` workers; there is no scenario
where one is present without the other. The separate package would also require
`check_isolation.py` to be updated to include the new package, adding maintenance
surface for no security benefit.

### Threads instead of subprocesses (would have affected placement)

If Workers were threads rather than subprocesses, the Agent would be a class within
`dropchannel-node` with no new package question. This was evaluated and rejected
independently: the dominant Worker failure mode is a hung blocking storage call,
which cannot be forcibly terminated from within a thread. Subprocesses allow SIGKILL
on a stuck Worker without collateral damage. See `history/v0.9.md` for the full
thread vs. subprocess analysis.

---

## Consequences

- `dropchannel-node/pyproject.toml` gains one script entry point (`dc-agent`).
- `check_isolation.py` requires no changes.
- The `agent/` submodule is subject to the same crypto isolation constraint as the
  rest of `dropchannel-node` without additional configuration.
- Future contributors should understand that `dropchannel-node` contains both the
  Worker-level protocol logic (in `protocol/`) and the Agent-level supervisor logic
  (in `agent/`). The package name reflects the Worker role; the Agent is the
  host-level coordinator of Workers.
