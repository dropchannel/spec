# ADR-0039: Branch Naming Convention

**Status:** Accepted  
**Deciders:** Paul Czajka  
**Date:** 2026-03-27
**Governs:** All repositories under github.com/dropchannel

---

## Context

The DropChannel project spans multiple repositories across the `dropchannel` GitHub org. As the number of repos and contributors grows, inconsistent branch naming creates friction: it is harder to understand the nature of in-flight work at a glance, harder to associate branches across repos when a single decision mandates changes in several places, and harder to enforce any review or automation policy without a predictable naming surface.

No branch naming convention existed prior to this decision.

---

## Decision

Branches across all DropChannel repositories follow the format:

```
{type}/{description}
```

`type` is one of: `feat`, `fix`, `refactor`, `docs`, `chore`, `adr`.

`description` is lowercase and kebab-case. Length is left to author judgment.

`main` is the protected trunk in all repositories. All substantive work is done on a branch and merged via pull request. Minor non-functional changes — documentation corrections, typos — may be committed directly to `main` at the author's discretion. This laxity is expected to tighten as the project matures.

Branches are short-lived and should be deleted after merging.

---

## Alternatives Considered

`{type}/{scope}/{description}` — a three-segment format that includes a scope between type and description. This is common in monorepos where a scope disambiguates which package or app is being touched. Ruled out because in a multi-repo org each repo already provides that context; the scope segment adds ceremony without payoff.

`{repo}/{type}/{description}` — encoding the repo name in the branch. Ruled out because branches exist within a repo, making the repo segment redundant by definition.

Conventional commit types only (`feat`, `fix`, `chore`, `refactor`, `docs`) without a domain-specific `adr` type — ruled out because ADR authorship is a well-defined, frequent work pattern in this org that maps poorly onto the standard types. `adr` as a first-class type makes governance branches immediately legible and enables cross-repo tracking of ADR implementation work.

---

## Reasoning

`{type}/{description}` is the most widely recognized branch naming pattern in open source and the format most contributors will arrive already knowing. Staying close to convention lowers onboarding friction.

The `adr` type earns its place because an ADR in `dropchannel/` may mandate changes across multiple repos. Naming the corresponding implementation branch `adr/{description}` in each affected repo creates a consistent, scannable signal: when all branches of that name have merged, the ADR's implementation is complete. This pattern is not available if ADR work is folded into `chore` or `docs`.

Omitting issue numbers keeps branch names short and removes a dependency on any particular issue tracker. The project does not currently use one, and the convention should not assume one.

The laxity on direct `main` commits for minor non-functional changes reflects the current scale of the project. Recording the expectation that this will tighten means the future tightening is a refinement, not a reversal.

---

## Consequences

- `governance/conventions/branching.md` is created in the `dropchannel` repository as the living reference for this convention.
- All new branches across all repositories should follow the format from the date this ADR is accepted.
- `[Deferred]` Enforcement via branch protection rules or CI is not defined here and is left to a future ADR or convention update.
- `[Deferred]` Commit message conventions are not addressed here and warrant a separate decision.
