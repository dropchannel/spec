# ADR-0037: Unified Vocabulary and Protocol Renaming

**Status:** Accepted  
**Date:** 2026-03-19  
**Deciders:** Paul Czajka  
**Repos affected:** `dropchannel`, `winch-protocol`, `conveyer-protocol`, `piston-protocol`, `dropchannel-py`, `.github`

---

## Context

The DropChannel system accumulated terminology organically across its first several
protocol versions. As the system matured and the protocol registry expanded, two
categories of friction emerged.

### Structural terminology friction

The word **"slot"** was overloaded across two distinct concepts: the directional conduit
between endpoints (e.g. `a-to-b`) and the physical storage position within a backend.
These are meaningfully different things ŌĆö one is a logical address, the other is a
storage container ŌĆö but they shared a name, causing confusion in specs, code, and
conversation.

The word **"channel"** was similarly overloaded. At the product level, "DropChannel"
implies a channel is the named thing you drop a payload into. At the implementation
level, `channel_id` was a parameter to every storage operation, creating a
name-within-name collision: "I'm using DropChannel, and I'm passing a `channel_id` to
a `ChannelProvider` to write to a `slot`." None of those terms mapped cleanly to the
conceptual model a new contributor or user would arrive with.

The term **"Node"** imported assumptions from distributed systems (routing intelligence,
network participation, state awareness) that conflicted with DropChannel's model, where
forwarding participants are crypto-blind, stateless between restarts, and explicitly
outbound-only. A Node implies agency. DropChannel's forwarding participants have none ŌĆö
they execute what the protocol dictates and nothing more.

### Protocol terminology friction

Protocol names (Winch, Conveyer, Ring, Piston) were mechanical metaphors with no shared
conceptual lineage. They did not compose or contrast in a way that aided understanding.
More importantly, protocol design had evolved: Winch and Piston converged into a single
richer bidirectional turn-passing protocol; and Ring was deprioritized.

The system needed a unified vocabulary rooted in a single coherent metaphor that would
scale as new protocols were added.

### The core inversion

A key insight drove the vocabulary decision: in DropChannel, the protocol is the mover,
not the forwarding participant. A Node has no independent agency ŌĆö it carries payloads
between ChannelProviders because the protocol dictates every action it takes. This is
the inverse of conventional distributed systems, where nodes are active agents making
routing decisions. DropChannel's forwarding participants are passive infrastructure.
The metaphor needed to reflect that inversion precisely.

---

## Decision

Adopt a **Waterway system** as the unified conceptual framework for all DropChannel
structural and protocol terminology. A Channel is a path between two fixed
endpoints. Waterways flow through it, each with different water characteristics governed by
their protocol. Docks line the banks. Rafts are passive ŌĆö carried by the water, not
self-propelled.

All existing terms are renamed accordingly and applied consistently across all repos,
specs, config schemas, code, and documentation.

---

## New Structural Vocabulary

### Concept definitions

| Old Term | New Term | Definition |
|---|---|---|
| Channel / `channel_id` (namespace) | **Channel** | The named path between two fixed endpoints. Defines the route. All Waterways within a Channel connect the same two endpoints. Independent lifecycle from any client. |
| Slot (as `a-to-b` conduit) | **Waterway** | An isolated flow path within a Channel. Protocol-typed via name prefix. Inherits the Channel's endpoints. Multiple Waterways with different protocols can coexist in one Channel. |
| Slot (as physical storage position) | *(retired)* | Eliminated. The Waterway is the container. Files live directly within Waterways. No intermediate concept needed. |
| ChannelProvider | **Dock** (ABC: `DockProvider`) | A storage instance ŌĆö GCS, Dropbox, local filesystem, httprelay. Sits on the bank. Serves all Waterways passing through the Channel. Purely physical ŌĆö no protocol or directional awareness. |
| Node | **Raft** | The passive mechanism that carries payload files between Docks along a Waterway. Crypto-blind. No independent agency ŌĆö behavior entirely determined by the Waterway's protocol. |

### The address model

A payload file is fully addressed by three coordinates:

```
(channel, waterway, filename)
```

- **Channel** ŌĆö the named path between two endpoints, using one or more specific Docks (e.g. `chat`)
- **Waterway** ŌĆö the protocol-defined message flow within the Channel (e.g. `riverway-bob`)
- **Filename** ŌĆö the specific payload file within the Waterway

The Dock instance is implicit ŌĆö it is whichever storage backend is being operated
against. Waterway names are invariant across all Docks. The filename within a Waterway may
change; the Waterway name never does.

### Dock interface

The `ChannelProvider` interface is renamed to `DockProvider`. Method signatures are
updated to reflect the new address model:

```python
# Primary operations
write(channel, waterway, filename, data) ŌåÆ bool      # False if filename already present
read(channel, waterway, filename) ŌåÆ bytes|None        # Destructive. Terminating endpoint only.
peek(channel, waterway, filename) ŌåÆ bytes|None        # Non-destructive.
exists(channel, waterway, filename) ŌåÆ bool            # Primary polling operation.
delete(channel, waterway, filename) ŌåÆ None            # Idempotent.

# Meta operations (heartbeat/observability only)
meta_write(channel, waterway, filename, data) ŌåÆ bool
meta_read(channel, waterway, filename) ŌåÆ bytes|None
meta_delete(channel, waterway, filename) ŌåÆ None
meta_list(channel, waterway) ŌåÆ list[str]
```

### Upper and Lower Dock semantics

A Waterway has a statically defined initiator endpoint at the **Upper** and a statically
defined responder endpoint at the **Lower**. These are fixed geographic properties of
the Waterway ŌĆö they never swap regardless of which direction the protocol is flowing at
any moment. Each protocol has statically-defined initiator-side endpoint, which
always corresponds to Upper.

From a Raft's perspective:

- `upper_dock` ŌĆö the Dock on the initiator's side
- `lower_dock` ŌĆö the Dock on the responder's side

The protocol state machine uses these as stable references. For Tideway, even-generation
files flow from Upper toward Lower; odd-generation files flow from Lower toward Upper.
For Riverway, flow is always Upper toward Lower .

### Config and runtime renames

| Old Term | New Term | Notes |
|---|---|---|
| `channel_id` (config field) | `channel` | |
| `slot` (config/code parameter) | `waterway` | |
| `recv_backend` | `upper_dock` | Dock on initiator's side. Fixed ŌĆö never swaps. |
| `send_backend` | `lower_dock` | Dock on responder's side. Fixed ŌĆö never swaps. |
| `node_id` | `raft_id` | Unique stable identifier for the Raft/host. |
| Package `dropchannel-node` | `dropchannel-raft` | Package rename. |
| Meta slot | **Meta Waterway** | Heartbeat/observability namespace within a Dock. |

Example Worker config after rename:

```toml
[workers.chat]
channel   = "chat"
waterway  = "tideway-bob"
upper_dock = "my-gcs"
lower_dock = "local-relay"
```

### Agent / Worker runtime

The **Worker** term is preserved as the runtime subprocess concept. **Raft** is the
protocol-level concept a Worker runs. This distinction is intentional: a Worker *runs*
a Raft. Collapsing them would conflate the runtime supervisor layer with the protocol
layer.

| Old Term | New Term |
|---|---|
| `node_id` in `[agent]` config | `raft_id` |
| `channel_id` in `[workers]` config | `channel` |
| `recv_backend` in `[workers]` config | `upper_dock` |
| `send_backend` in `[workers]` config | `lower_dock` |
| Worker `channel_id` in telemetry blob | `channel` |

### Terms that do not change

- **Agent** ŌĆö host-level supervisor, one per physical host
- **Worker** ŌĆö runtime subprocess; distinction from Raft is intentional
- **Endpoint** ŌĆö the terminating participant (Client A, Client B)
- **Relay** ŌĆö network-accessible service exposing a Dock over HTTP

### Key structural clarifications

**1. Channel defines the two endpoints.**
All Waterways within a Channel connect the same two endpoints. A Waterway cannot exist outside
its Channel's defined route. This is not a limitation ŌĆö it is the precise definition
of what DropChannel is: a point-to-point system between exactly two endpoints.

**2. Channel is a namespace, not a typed conduit.**
Protocol identity lives at the Waterway level. A Channel can contain Waterways of different
protocols simultaneously.

**3. Waterway name encodes protocol prefix and identity, not direction.**
`tideway-a-to-b` is valid for a unidirectional Waterway. `tideway-chat` is correct where
direction is protocol-state-dependent. Direction is never assumed from the Waterway name.

**4. Upper and Lower are fixed.**
`upper_dock` and `lower_dock` reflect stable geographic position on the Waterway. They do
not swap when the protocol reverses direction.

**5. Raft behavior is entirely protocol-delegated.**
The Waterway's protocol prefix determines which Raft implementation runs. That
implementation decides which Dock to operate against at any point in time. The Dock
has no opinion about direction.

**6. `(channel, waterway)` is the universal address.**
Every Dock operation uses this pair plus a filename. The Dock instance is the physical
location; the pair is the container within it.

## New Protocol Vocabulary

### Rationale for protocol renaming

Protocol names are renamed to align with the Waterway metaphor. Names are chosen to be:

- Purely nominal ŌĆö no verb/noun ambiguity
- Evocative of the natural water behavior that drives the protocol
- Clearly distinct from one another
- Consistent with the passive-Raft model ŌĆö the protocol is the mover, not the Raft

### Protocol registry

| Old Name(s) | New Name | Prefix | Characteristic |
|---|---|---|---|
| Winch, Piston | **Tideway** | `tideway-` | Turn-passing, fixed initiator, bidirectional on single Waterway |
| Conveyer | **Riverway** | `riverway-` | Continuous, unidirectional, overwrite-always, no ACK |
| Ring | *(dropped)* | ŌĆö | Deferred; no concrete requirement currently |

---

### Tideway Protocol

**Replaces:** Winch, Piston (unified)

**Name rationale:** A tide moves in one direction, then reverses ŌĆö governed by natural
forces rather than by the vessel. The reversal is structured, predictable, and governed
by who holds the payload (the turn token). One tide at a time, in one direction at a
time. The Raft doesn't choose its direction ŌĆö the tide determines it.

**Behavior:**

- One file in the Waterway at any time
- Fixed initiator at the Upper-side, fixed responder at the Lower ŌĆö established at Waterway creation
- Initiator is the sole party permitted to write to an empty Waterway ŌĆö prevents collision
- Responder may only write in response to receiving a payload
- Response payload travels back toward the initiator (Upper) through the same Waterway
- Return payload may be a minimal ACK blob or a full encrypted response
- Both forward and return payloads are AES-256-GCM encrypted
- If bidirectional initiation is required, two Tideway Waterways are used ŌĆö one per direction:

  ```
  tideway-a    ŌåÉ A is initiator (Upper), B is responder (Lower)
  tideway-b    ŌåÉ B is initiator (Upper), A is responder (Lower)
  ```

**Filename generation scheme:**

Rafts detect direction reversal via filename generation parity, without reading content:

```
{generation}-{opaque}
```

Examples:

```
0-7f3a9c2e1b    ŌåÉ generation 0: initiator payload, flowing toward Lower
1-4d8b1a9f3c    ŌåÉ generation 1: responder payload, flowing toward Upper
2-9a1d3e8b2f    ŌåÉ generation 2: initiator payload again
```

- **Even generation** ŌĆö initiator's payload, direction: Upper ŌåÆ Lower
- **Odd generation** ŌĆö responder's payload, direction: Lower ŌåÆ Upper
- **Opaque suffix** ŌĆö random or nonce-derived hex; ensures global uniqueness; reveals
  no content or timing information to storage-layer observers
- **Raft logic:** `exists()` / `peek()` ŌåÆ parse generation number from filename ŌåÆ
  if parity differs from last seen, reverse direction
- Rafts are stateless with respect to direction ŌĆö generation parity is the sole signal;
  no local state required across restarts

**Use for:** request/response interactions, command/result handoffs, deaddrop exchanges,
any interaction requiring confirmed turn-passing with encrypted response.

---

### Riverway Protocol

**Replaces:** Conveyer

**Name rationale:** A Riverway flows continuously through a Channel in one direction ŌĆö it
does not accumulate, does not wait for a receiver, and the latest state is always what
matters. The Raft doesn't decide when to move ŌĆö the Riverway carries it continuously.

**Behavior:**

- Unidirectional, producer-driven, always Upper ŌåÆ Lower
- Overwrite-always ŌĆö latest file replaces previous, no history retained
- No ACK, no backpressure ŌĆö producer never stalls
- Consumer uses `peek()` ŌĆö non-destructive; multiple consumers supported
- Raft deduplication via hash of last forwarded file ŌĆö prevents re-forwarding identical
  content
- Unconditional sleep between cycles

**Use for:** telemetry feeds, sensor data, status signals, observability side-channels,
any producer-driven flow where only the latest value matters and individual entry loss
is acceptable.

---

### Ring Protocol ŌĆö Deferred

Ring (rolling window, ordered history, consumer cursor tracking) is dropped from the
active protocol registry. No concrete use case currently requires its specific semantics.
The protocol may be specified in a future revision if requirements emerge.

## Open Items

**1. `telemetry-` prefix migration**
The existing `telemetry-` prefix dispatches to Conveyer (now Riverway) behavior. The
prefix is functional rather than protocol-named. A decision is needed before execution:
retain `telemetry-` as a named exception in the protocol registry, or migrate to
`riverway-telemetry-*`. Retaining avoids a breaking change to existing deployments;
migrating achieves full consistency.

**2. Tideway filename scheme ŌĆö traffic analysis consideration**
Generation numbers in filenames expose exchange count to storage-layer observers. An
observer can determine how many exchanges have occurred on a Waterway without reading content.
This is an accepted tradeoff but must be explicitly noted in the Tideway protocol spec as
a known traffic analysis surface.

**3. `list()` ŌĆö deferred**
A `list(channel, waterway)` operation returning filenames within a Waterway is identified as a
future addition to the DockProvider interface. Deferred to a separate ADR. No
implementation changes required at this time.

**4. `upper_dock` / `lower_dock` role semantics on first startup**
The Tideway protocol spec must define how a Raft determines which direction to move on
first startup or restart, given only `upper_dock`, `lower_dock`, and the current Waterway
state at each Dock.

---

## Consequences

### Positive

- Single coherent Waterway metaphor across all structural and protocol terminology
- The passive-Raft model is accurately reflected in naming ŌĆö the protocol moves, the
  Raft is carried
- `upper_dock` / `lower_dock` provide stable, direction-independent Dock references that
  work correctly for both unidirectional (Riverway) and bidirectional (Tideway) protocols
- Waterway as a named container with files maps directly to how all current Dock backends
  actually work (GCS objects under a prefix, files in a directory)
- Protocol names (Tideway, Riverway) are clearly distinct, purely nominal, and evocative
  of the natural force that drives each protocol's behavior
- Dropping Piston and Ring reduces registry complexity without losing coverage of
  concrete use cases
- Channel-defines-endpoints is now explicit, accurately reflecting DropChannel's
  point-to-point constraint

### Negative / costs

- Broad rename surface: every spec file, all code in `dropchannel-py`, config schemas,
  ADR backlog, README files, and repo names require updates
- `dropchannel-py` version bump required to reflect interface changes (parameter
  renames throughout DockProvider)
- Existing deployments using `channel_id` / `slot` / `recv_backend` / `send_backend`
  config keys will require config migration
- Existing `winch-` and `conveyer-` prefixed Waterway names in deployments will require
  migration to `tideway-` and `riverway-` prefixes
